# 🔒 Row-Level Security (RLS) 最佳實踐指南 在 `node-pg-trigger` 的即時推送架構中，PostgreSQL 的 Row-Level Security (RLS) 是確保數據安全的核心。本指南將說明如何在生產環境中建立嚴謹的安全策略。 --- ## 🌟 核心機制：身分注入流程 1. **前端身分識別**: 使用者在 SDK 呼叫 `sdk.auth('user_id_or_token')`。 2. **tRPC Context 轉發**: 後端接收並驗證身分，將其存儲在連線 Context 中。 3. **資料庫 Session 綁定**: 在執行查詢或處理訂閱前，系統會執行 `SET LOCAL "request.user_id" = '...'`。 4. **RLS 策略生效**: 資料庫根據 Session 中的 `request.user_id` 動態過濾每一行紀錄。 --- ## 🛠️ 生產環境常用策略範例 ### 1. 私有數據保護 (僅擁有者可讀寫) 適用於個人設定、私有筆記等。 ```sql -- 啟用 RLS ALTER TABLE user_profiles ENABLE ROW LEVEL SECURITY; -- 建立策略：僅當 id 符合目前使用者時允許所有操作 CREATE POLICY profile_owner_policy ON user_profiles FOR ALL TO authenticated USING (id::text = current_setting('request.user_id', true)); ``` ### 2. 共享協作模式 (團隊成員可讀) 適用於項目管理、群組聊天等。 ```sql ALTER TABLE project_tasks ENABLE ROW LEVEL SECURITY; -- 建立策略：檢查使用者是否在該項目的成員名單中 CREATE POLICY project_member_policy ON project_tasks FOR SELECT USING ( EXISTS ( SELECT 1 FROM project_members WHERE project_id = project_tasks.project_id AND user_id::text = current_setting('request.user_id', true) ) ); ``` ### 3. 公開數據但受限寫入 (唯讀公眾，擁有者可更) 適用於部落格文章、公開評論。 ```sql ALTER TABLE blog_posts ENABLE ROW LEVEL SECURITY; -- 公開讀取 CREATE POLICY public_read_policy ON blog_posts FOR SELECT USING (status = 'published'); -- 僅擁有者可更新 CREATE POLICY owner_update_policy ON blog_posts FOR UPDATE USING (author_id::text = current_setting('request.user_id', true)); ``` --- ## ⚡ 效能優化建議 * **索引優化**: 確保 RLS `USING` 子句中引用的欄位（如 `user_id`, `project_id`）具備索引，否則大量併發訂閱時會造成 CPU 負載升高。 * **避免複雜子查詢**: 盡量使用簡單的欄位比對。若需跨表檢查權限，考慮使用緩存或在應用程式層預先過濾。 * **預設拒絕**: 建議將 `ALTER TABLE ... FORCE ROW LEVEL SECURITY` 設為常態，防止管理員權限意外洩漏數據。 --- ## 🎓 總結 RLS 讓 `node-pg-trigger` 具備了「數據庫即防火牆」的能力。開發者不需在 Node.js 中編寫複雜的過濾邏輯，只需在 SQL 層定義好規則，即可確保即時推送的絕對安全。
